# Архитектура приложения «Забор» (Техническая документация)

## 1. Общий обзор системы
- **Назначение приложения:** Desktop-приложение «Забор» представляет собой сервис для социальных коммуникаций и голосового общения, являющийся легковесным аналогом платформ вроде Discord или TeamSpeak. 
- **Основные бизнес-функции (User Stories):**
  - Регистрация, авторизация и управление профилем (в т.ч. управление кастомными аватарами и гифками).
  - Создание и администрирование пользовательских голосовых каналов.
  - Friend-система (список друзей, добавление/удаление, статусы в сети).
  - Голосовые звонки (P2P, каналы) с активным WebAssembly шумоподавлением.
  - Гибкая настройка аудио: выбор устройств I/O, регулировка громкости каждого участника.
  - Геймификация: система скрытых достижений/«ачивок».
  - Встроенная система администрирования.

---

## 2. Стек технологий
- **Frontend:**
  - `React 18` + `TypeScript`.
  - Вёрстка и стилизация: `TailwindCSS` + функциональные UI-компоненты.
  - Иконки: `lucide-react`.
- **Desktop Environment:**
  - `Electron` (сборка через `electron-vite` и `electron-builder`), целевая платформа: Windows (`NSIS`).
- **Голосовой движок:**
  - `WebRTC` для прямого P2P общения + `@jitsi/rnnoise-wasm` для локального подавления шумов.
- **Backend:**
  - `C#` / `.NET 10.0` Web API.
  - Двусторонний обмен данными: `ASP.NET Core SignalR`.
- **База данных:**
  - `SQLite` (взаимодействие через `Entity Framework Core`).
- **Инфраструктура:**
  - Backend размещается на Linux сервере и работает через обратный прокси. Сборка фронтенда происходит прозрачно через встроенные `.bat`/`.cmd` скрипты.

---

## 3. Структура Frontend
- **Архитектура каталогов:** 
  Базируется на классическом паттерне Electron-Vite:
  - `src/main` — системные процессы ОС, инициализация окон, IPC.
  - `src/preload` — безопасная прослойка (IPC Bridges).
  - `src/renderer/src` — исходный код React-приложения.
- **Управление состоянием (Global State Management):** 
  Реализовано на базе библиотеки **Zustand** (`store/useAppStore.ts`). Состояние полностью атомарно: хранит сессию, список каналов, друзей, списки активных голосовых сессий (voice users), что избавляет приложение от лишних рендеров, типичных для Context API или Redux.
- **Маршрутизация (Routing):** 
  Одностраничная SPA-архитектура (без библиотек типа react-router). Экраны и модальные окна переключаются через условный рендеринг непосредственно внутри `App.tsx` (состояния `activeTab`, `authStep`).
- **Основные компоненты и UI-кит:** 
  Лежат в `src/renderer/src/components`. Используется Material Design 3 эстетика (`Md3Slider`, `Md3Switch`, `AvatarImg`, кастомный `TitleBar`). Обильно используется Glassmorphism.
- **Взаимодействие с API / Backend:** 
  Спрятано под фасадом синглтон-сервисов:
  - `services/signalr.ts` — инкапсулирует RPC вызовы на бекэнд.
  - `services/webrtc.ts` — управляет аудио-потоками и Peer-соединениями.

## 4. Структура Backend
- **Архитектура кода:** 
  Компактная монолитная архитектура, разделённая на два .NET проекта:
  - `Zabor.Core` — Классы доменных моделей и общие контракты (`User.cs`, `VoiceChannel.cs`, `Messages.cs`).
  - `Zabor.Server` — Основной запускаемый проект (точка входа `Program.cs`, слой данных `AppDbContext`, слой сервисов типа `UserService.cs` и точка доступа к WebSocket `ZaborHub.cs`).
- **Модели данных (ER-структура):**
  ```text
  Entity: User
   - Id: string (PK)
   - DisplayName, Username, PasswordHash
   - AvatarBase64, AvatarColor
   - AudioSettingsJson (TEXT)       // Настройки звука (JSON)
   - AchievementsJson (TEXT)        // Достижения (JSON)
   - IsServerMuted / IsServerDeafened (Int/Bool)

  Entity: VoiceChannel
   - Id: string (PK)
   - Name: string
   - OwnerId: string (FK -> User)
  ```
- **Ключевые Endpoints:**
  - Большинство маршрутизации отсутствует. Сервер экспонирует REST-заглушки проверки статуса (`/`, `/joke`) и обслуживает основную транспортную магистраль: **SignalR Hub Endpoint:** `/zabor_v3`.
- **Аутентификация и авторизация:** 
  Используется кастомная аутентификация поверх SignalR-соединений (маппинг `ConnectionId` на сущность пользователя в памяти). Сессия локально кэшируется на клиенте (через Electron Native `loadSession() / saveSession()`).
- **Обработка ошибок и логирование:** 
  Используются консольные логи `Console.WriteLine` и встроенный SignalR Exception Handling. Миграции БД SQLite выполняются raw-инъекциями (`ExecuteSqlRaw`) на старте приложения (`Program.cs`).

## 5. Интеграции и взаимодействие
- **Протоколы общения Frontend-Backend:** 
  Обмен происходит через WebSockets с резервированием до Long Polling посредством библиотеки `@microsoft/signalr`. 
- **Voice-коммуникация:** 
  Не использует сторонние медиа-сервера (TURN/STUN разворачиваются при необходимости конфигом). Голос передаётся **строго P2P (Peer-to-Peer) по протоколу WebRTC** в топологии Mesh (каждый участник канала соединен с каждым напрямую).

---

## 6. Data Flow: от действия до БД и голоса
**Пример сценария: подключение пользователя к голосовому каналу.**

1. **Frontend Trigger:** Пользователь жмёт на канал в интерфейсе (Component).
2. **Local State:** Zustand устанавливает флаг прогресса загрузки и ожидаемого канала `setPendingChannelSwitch`.
3. **Обмен с Backend:** Фронтенд вызывает RPC-метод SignalR `joinChannel(channelId)`. Ожидает ответа сервера.
4. **Backend Processing:** `ZaborHub` получает запрос, валидирует лимит пользователей в комнате, добавляет `ConnectionId` в SignalR `Group` канала, обновляет статус в `AppDbContext` и асинхронно рассылает всем участникам группы событие `UserJoined`.
5. **Апдейт UI:** Клиентские приложения получают broadcast, `signalr.ts` дёргает экшены в Zustand для обновления списка юзеров (`store.voiceUsers`). UI немедленно отрисовывает новую аватарку в карточке канала.
6. **WebRTC Handshake:** 
   Слой `webrtc.ts` обнаруживает нового пользователя. Он генерирует SDP-оффер (предложение связи) и отправляет его через SignalR (как сигнальный сервер).
7. **P2P канал установлен:** Клиенты обмениваются ICE-кандидатами через SignalR. WebRTC устанавливает прямое UDP/TCP соединение. 
8. **Медиапоток:** Аудио с микрофона проходит через модуль `rnnoise-wasm` на фронтенде и транслируется напрямую собеседникам, минуя бэкенд.
